### 1.2 Ashby: не смешивать валюты между compensationTiers

**Название:** По-тировый парсинг Ashby-компенсаций · **S** · импакт 4 · судьи: recruiter 9, solo-maintainer 9

**Шаги.**

3. В `normalize_salary` добавить защиту: `max/min > 20` → сброс в dropped_oob с отдельным счётчиком (JPY-канарейка).
4. Тест в `tests/test_salary.py`: два тира USD+JPY по мотивам реальной строки → берётся только USD-тир; тест на счётчик ratio>20.

### 1.3 Greenhouse-зарплаты: якорная пара «$X – $Y» вместо «первых двух токенов»

**Название:** Переписать _parse_greenhouse + починить _PAIR (запятые, суффикс, гарды) · **S** · импакт 5 · судьи: recruiter 9, solo-maintainer 9, cis-dev 6. Объединяет три идеи пула (якорение, _PAIR-запятые, плаузибилити-гард) — делать одним PR.

**Зачем.** GH — 4 290 вакансий (65% датасета), а `_parse_greenhouse` берёт «первые два $-токена ≥1000» из всего текста: amplitude — 5 вакансий с вилкой $2,000–$190,000 (бонус/401k как min), databricks — 14 строк с max $1,000,000 (OTE-текст), cloudflare $198k–$2M; текст «We raised $100,000,000… Salary range: $150,000-$180,000» теряет честную вилку. Параллельно `_PAIR` (строка 21) не знает запятых — воспроизведено: `_parse_summary('$150,000 - $180,000') → None`, а «$150-200K» даёт min=150 долларов; класс `[-–—to]+` матчит бессмысленные «ot»/«tt».

**Шаги.**
1. `_PAIR`: группа числа `(\d{1,3}(?:,\d{3})+|[\d.]+)`, разделитель — явная альтернатива `\s*(?:-|–|—|to|до)\s*`; пропагация суффикса: если у hi есть K/M, а lo < 1000 без суффикса — умножить lo на тот же множитель.
2. `_parse_greenhouse`: искать в тексте СМЕЖНЫЕ пары через обновлённый `_PAIR` в окне ±120–150 символов от якорных слов `(salary|base pay|compensation|annual|range)`; одиночные $-суммы игнорировать.
3. Гарды до усреднения: отбраковать пару при `max/min > 3` или `min < SALARY_MIN_USD/2`, оба числа в [20k, 1.5M]; счётчик отклонённых пар — в `data/cache/last_run.json`.
4. Golden-тесты в `tests/test_salary.py` дословно из данных (обязательная часть приёмки): amplitude ($2k-бонус игнорируется), databricks ($1M-OTE отбраковывается), «We raised $100,000,000… Salary range: $150,000-$180,000» → (150000, 180000), «$150,000 - $180,000», «$150-200K» → (150000, 200000).
5. Локально прогнать aggregate и сравнить медианы по компаниям до/после (databricks должен упасть с $570k-хвоста) — дельта медианы databricks обязательна в описании PR.
6. **Приёмка:** golden-тесты amplitude/databricks зелёные; `duckdb`-запрос по свежему снапшоту: GH-строк с `max/min > 3` ≤ 5 (сейчас 31 из 2 172 зарплатных GH-строк; по всем источникам — 57). Прежний порог «доля <2%» не годится как приёмка: он выполнен уже ДО фикса (31/2172 ≈ 1.4%) и пропустил бы PR без единого изменения кода. Все 29+ тестов зелёные.

**Не сломай.** `_parse_summary` (Ashby-фоллбэк) использует тот же `_PAIR` — прогнать его тесты. Не расширять здесь на £/€ — это отдельный шаг 1.9 тем же PR или следующим.

### 1.10 Надёжность pipeline.yml: volume-guard + честный push-retry

**Название:** Не коммитить «отравленный» день и не глотать упавший пуш · **M** · импакт 4-5 · судьи: recruiter 8, solo-maintainer 8 (обе части). Один PR по pipeline.yml + cli.

**Зачем.** Два тихих сценария потери/порчи данных. (1) Canary в pipeline.yml (строки 50-67) проходит при ЛЮБОМ одном живом эндпоинте, а `iter_jobs` в `etl/fetch.py` глотает ошибки досок: полный отказ Greenhouse (66% датасета; 429-шторм по IP Actions — реалистично) закоммитит снапшот из ~2 250 строк, и timeseries навсегда покажет фейковый обвал рынка. (2) Цикл `for i in 1 2 3; do git push && break || git pull --rebase; done` (строки 85-87) возвращает код ПОСЛЕДНЕЙ команды: три упавших push + успешный pull = зелёный шаг, день молча потерян, до 120 Gemini-вызовов сожжено впустую.

**Шаги.**
1. В `etl/cli.py::run()` расширить `data/cache/last_run.json` полем `by_source` из уже посчитанного Counter.
2. Перед `write_partition` посчитать 7-дневную медиану строк по source из `data/snapshots/*` (тем же duckdb, что в `existing_job_uids()`); если любой источник <50% медианы или сумма <60% — `sys.exit(2)` ДО записи партиций (пороги щедрые — совет solo-maintainer). Шаг «Alert on failure» создаст issue; второй cron 7:47 — автоматический ретрай.
3. Первые 7 дней истории медиана считается по доступным дням; при <3 днях истории guard пропускать (иначе заблокирует себя на старте).
4. В pipeline.yml после retry-цикла: `git push || { echo 'push failed after retries'; exit 1; }`. Тот же фикс — в backfill.yml (строка ~37).
5. Тест на guard в `tests/test_cli_dry.py` на синтетике (источник упал до 30% медианы → exit 2, партиции не записаны).
6. **Приёмка:** локальный dry-тест guard'а зелёный; в pipeline.yml ручная проверка: `bash -n` на изменённый скрипт-блок; симуляция «push падает» (например, невалидный remote в тест-ветке) даёт красный шаг.

**Не сломай.** Идемпотентность и guard по существующему dt (§8) — volume-guard срабатывает ДО write_partition, полупустые партиции не появляются. `[skip ci]` не использовать (§8). Секреты не добавлять.
